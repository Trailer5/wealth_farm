# 项目开发规则

## 定位

本文件记录项目开发的基本规则、项目组织方式和 Agent 协作约束。
本文件记录长期有效的规则。本文件不记录临时想法。
本文件必须使用明确的肯定句和否定句。

## 项目说明

`README.md` 用于介绍项目、说明项目用途、记录安装方法、运行方法和使用方法。
`README.md` 不记录开发过程细节。`README.md` 不记录长期规划、模块内部设计或临时任务清单。

## 文档目录

所有开发文档必须落入 `docs/`。
`docs/long_term/` 记录长期计划、长期路线图和长期技术债。
`docs/modules/` 记录每个模块的要求、边界、能力、输入、输出和依赖。
直接落在 `docs/` 下的文档只记录短期开发计划、当前阶段说明和跨模块临时协调事项。
开发文档不得散落在 `src/`、`data/`、`logs/`、`references/` 或 `temp/` 中。

## 代码与测试目录

代码、脚本和可执行逻辑必须落入 `src/`。
单元测试必须落入 `src/unit_tests/`。
`src/` 不存放长期规划文档。`src/` 不存放业务数据、数据库文件、日志文件、外部项目或临时文件。

## 数据目录

数据记录、数据样本、数据库文件和 SQLite 文件必须落入 `data/database/`。
`data/` 不存放源码。`data/` 不存放开发规则。`data/` 不存放临时调试输出。

## 日志目录

运行日志、调试日志和审计日志必须落入 `logs/`。
`logs/` 不存放业务源码。`logs/` 不存放长期文档。`logs/` 不存放数据库主文件。

## 外部项目目录

外部项目、第三方代码快照、参考仓库和外部资料必须落入 `references/`。
`references/` 中的内容默认只读。Agent 不得修改外部项目内容，除非用户明确要求。
`references/` 不作为本项目源码目录。项目运行代码不得依赖未说明来源和版本的外部快照。

## 临时目录

临时文件、一次性中间产物、实验输出和可删除缓存必须落入 `temp/`。
`temp/` 中的内容不得作为长期事实来源。`temp/` 中的内容不得作为项目必需输入。

## 协作边界

Agent 是项目协作者。Agent 必须保持前后约束一致。
Agent 必须保护用户已有改动。Agent 不得回滚、删除或重写与当前任务无关的内容。
Agent 必须标注不确定事项。Agent 不得把猜测写成事实。

## 版本兼容

用户确立某一版为生产规范、milestone 或正式版之前，项目不需要兼容历史格式和历史接口。
未定版阶段必须优先直接修正设计。未定版阶段不得为了兼容旧方案增加兼容层。

## 一致性要求

新增目录必须符合本文件定义的职责。
新增文档必须放入正确的文档层级。
新增代码必须放入 `src/`。
新增测试必须放入 `src/unit_tests/`。
新增数据必须放入 `data/`。
新增日志必须放入 `logs/`。
新增外部参考必须放入 `references/`。
新增临时文件必须放入 `temp/`。
任何规则变更必须先修改本文件，再同步调整相关入口文件和说明文档。
